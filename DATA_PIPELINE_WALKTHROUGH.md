# Data Pipeline Walkthrough

This project implements a trading data lakehouse on AWS using Airflow, EMR Serverless, Spark, AWS Glue Catalog, S3, Athena, and Apache Iceberg. Data moves from raw S3 files into Bronze Iceberg tables, then into typed and quality-gated Silver tables, then into analytics-ready Gold aggregates.

At a high level:

1. Raw trade and quote ticks arrive in S3 under hourly partitions.
2. Raw corporate-event CSVs arrive once per weekday.
3. Airflow DAGs submit PySpark jobs to EMR Serverless.
4. Spark reads raw S3 files, MERGEs them into Bronze Iceberg tables, transforms and validates them into Silver, and computes Gold aggregates.
5. Athena row-count checks validate Gold outputs.
6. A weekly maintenance DAG compacts and cleans all Iceberg tables.

All Spark sessions are configured to use UTC, the Glue Iceberg catalog, and an S3 warehouse at `s3://<BUCKET>/warehouse`.

## Pipeline Summary

```text
raw/events/events_<DATE>.csv
  -> bronze.events_raw
  -> silver.events
      -> silver.trades event enrichment
      -> gold.event_windowed_volume

raw/trades/dt=<DATE>/hh=<HOUR>/*.jsonl.gz
  -> bronze.trades_raw
  -> silver.trades
      -> gold.ohlcv_1m
      -> gold.vwap_daily
      -> gold.event_windowed_volume

raw/quotes/dt=<DATE>/hh=<HOUR>/*.jsonl.gz
  -> bronze.quotes_raw
  -> silver.quotes
      -> gold.bid_ask_spread_1m
```

## Raw Data

The raw input data is synthetic market data generated by `generator/generate_synthetic.py` and uploaded directly to S3. The generator emits data only for weekdays in the requested date range and writes three raw feeds:

```text
s3://<bucket>/raw/trades/dt=YYYY-MM-DD/hh=HH/part-<uuid>.jsonl.gz
s3://<bucket>/raw/quotes/dt=YYYY-MM-DD/hh=HH/part-<uuid>.jsonl.gz
s3://<bucket>/raw/events/events_YYYY-MM-DD.csv
```

The simulated exchanges are:

```text
NYSE, NASDAQ, ARCA
```

The symbol universe is:

```text
AAPL, MSFT, AMZN, GOOGL, TSLA, NVDA, META, JPM, BAC, GS,
ORCL, CRM, AMD, INTC, NFLX, DIS, WMT, XOM, CVX, PFE
```

Trade and quote timestamps are generated in UTC. The generator uses active hours 09:00 through 15:59 UTC, with the 09:00 hour beginning at 09:30 UTC. This matches the downstream convention that S3 `dt=` and `hh=` partitions are UTC partitions.

## Synthetic Data Generation Process

The synthetic generator creates an independent weekday market simulation for each requested calendar date. For each date, it loops through every exchange, every symbol, and each active UTC hour. The generated data is not intended to model a full exchange order book; it is a compact tick-level process that produces internally consistent trades, quotes, and event windows for exercising the lakehouse pipeline.

### Core Parameters

| Parameter | Value |
| --- | --- |
| Exchanges | `NYSE`, `NASDAQ`, `ARCA` |
| Symbols | 20 large-cap symbols listed above |
| Active hours | 09:00-15:59 UTC, with the first hour starting at 09:30 UTC |
| Tick inter-arrival time | Exponential distribution with mean 0.3 seconds per `(exchange, symbol)` stream |
| Bid/ask spread | 5 basis points around the current latent fair value |
| Trade probability per tick | 60% |
| Trade side probabilities | `BUY`: 45%, `SELL`: 45%, `MID`: 10% |
| Trade size | Discrete uniform integer from 1 to 999 shares |
| Quote bid/ask size | Discrete uniform integer from 100 to 4,999 shares |
| Price innovation | Log-normal random walk with per-tick normal shock sigma of 0.0003 |
| Events per day | Poisson distribution with mean 3, capped at the number of symbols |

Each symbol starts the day from a fixed initial reference price. For example, AAPL starts at 185.0, MSFT at 380.0, AMZN at 175.0, GOOGL at 140.0, TSLA at 250.0, and NVDA at 480.0. The price state is tracked by symbol and carried forward as that symbol is generated across hours and exchanges.

### Tick, Trade, and Quote Simulation

For a given `(exchange, symbol, date, hour)` stream, the simulation starts with a latent fair value `P_0` and an initial prevailing quote:

```text
spread_0 = P_0 * 5 / 10,000
bid_0    = P_0 - spread_0 / 2
ask_0    = P_0 + spread_0 / 2
```

The stream then advances through irregular tick times. The waiting time between ticks is sampled from an exponential distribution:

```text
delta_t ~ Exponential(mean = 0.3 seconds)
```

At each tick timestamp, the generator performs the following sequence:

1. It optionally emits a trade with probability 0.6.
2. If a trade occurs, the trade side is sampled as `BUY`, `SELL`, or `MID`.
3. The trade price is set from the quote that was already prevailing before this tick:
   - `BUY` executes at the previous ask.
   - `SELL` executes at the previous bid.
   - `MID` executes at the midpoint of the previous bid and ask.
4. The latent fair value advances by a log-normal step:

```text
epsilon_t ~ Normal(0, 0.0003)
P_t       = round(P_{t-1} * exp(epsilon_t), 4)
```

5. A new 5-basis-point quote is generated around the updated latent fair value:

```text
spread_t = round(P_t * 5 / 10,000, 4)
bid_t    = round(P_t - spread_t / 2, 4)
ask_t    = round(P_t + spread_t / 2, 4)
```

6. The new quote becomes the prevailing quote for the next tick.

This order matters. Trades are priced from the previously available quote, and the new quote is emitted only after the latent fair value moves. That avoids a lookahead pattern where a trade could execute against a quote update that had not yet existed at the time of the trade.

The sequence number starts from a timestamp-derived integer for the beginning of the hour and increments by one per tick. Quote records are emitted at every tick. Trade records are emitted only on ticks where the Bernoulli trade trial succeeds.

### Daily Event Simulation

Corporate events are generated once per weekday using a separate random stream from the trade and quote process. The number of events for the day is sampled as:

```text
N_events ~ Poisson(lambda = 3)
```

The count is capped at the number of available symbols, because the generator chooses symbols without replacement. This means a symbol appears at most once in the event calendar for a given day.

For each event:

1. A symbol is sampled from the symbol universe without replacement.
2. An event type is sampled from:

```text
EARNINGS, DIVIDEND, SPLIT, GUIDANCE, BUYBACK
```

3. The event start time is sampled uniformly by minute between 10:30 and 15:00 UTC.
4. The event duration is sampled as a discrete uniform integer from 15 to 60 minutes.
5. The event end time is computed as `event_start + duration`.

The 10:30-15:00 UTC start-time range is deliberate: it leaves at least one hour of simulated trading data before and after each event during the generated 09:30-16:00 UTC trading window. That gives the Gold event-window table enough data to compute pre-event and post-event volume.

The event feed is uploaded once per generated weekday as `raw/events/events_YYYY-MM-DD.csv`. Even days with zero sampled events still produce a CSV with only the header, so the Airflow sensor can distinguish "no events today" from "the event file never arrived."

### Raw Feed Records

The generator writes three raw feeds. The generation process is described above; this section is only a reference for the record shapes that land in S3.

| Feed | S3 path | Fields |
| --- | --- | --- |
| Trades | `raw/trades/dt=YYYY-MM-DD/hh=HH/part-<uuid>.jsonl.gz` | `exchange`, `symbol`, `exchange_ts`, `price`, `size`, `sequence_num`, `side`, `trade_id` |
| Quotes | `raw/quotes/dt=YYYY-MM-DD/hh=HH/part-<uuid>.jsonl.gz` | `exchange`, `symbol`, `exchange_ts`, `bid`, `bid_size`, `ask`, `ask_size`, `sequence_num` |
| Events | `raw/events/events_YYYY-MM-DD.csv` | `event_id`, `symbol`, `event_type`, `event_date`, `event_start`, `event_end`, `description` |

## DAGs

### `events_calendar_daily`

File: `dags/events_calendar_daily.py`

This DAG loads the corporate events calendar. It runs before the trading DAG because Silver trades join to `silver.events` for event enrichment.

Schedule:

```text
30 5 * * 1-5
```

That means Monday through Friday at 05:30 UTC. `catchup=False`, `max_active_runs=1`.

Required Airflow variables:

| Variable | Meaning |
| --- | --- |
| `BUCKET` | S3 bucket that stores raw data, jobs, warehouse data, logs, and Deequ assets. |
| `EMR_SERVERLESS_APP_ID` | EMR Serverless application ID. |
| `EMR_JOB_ROLE_ARN` | IAM role used by EMR Serverless jobs. |

Flow:

```text
events_sensor
  -> ingest_bronze_events
  -> bronze_to_silver_events
```

Tasks:

| Task | Type | Computes / touches |
| --- | --- | --- |
| `events_sensor` | `S3KeySensor` | Waits for `s3://<BUCKET>/raw/events/events_<DATE>.csv`. |
| `ingest_bronze_events` | EMR Serverless Spark job | Reads the daily events CSV and MERGEs rows into `bronze.events_raw`. |
| `bronze_to_silver_events` | EMR Serverless Spark job with PyDeequ | Reads `bronze.events_raw`, casts and validates events, then MERGEs into `silver.events`. |

The DAG uses `data_interval_end | ds` as the processing date, so the `DATE` passed to Spark is the calendar date on which the 05:30 UTC run fires.

### `trading_pipeline`

File: `dags/trading_pipeline.py`

This is the intraday trading pipeline. It ingests trades and quotes every 15 minutes during the configured weekday window, promotes them to Silver, and computes intraday Gold aggregates.

Schedule:

```text
*/15 6-22 * * 1-5
```

That means every 15 minutes, Monday through Friday, from 06:00 through 22:45 UTC cron slots. The code comments describe it as the trading session window. `catchup=False`, `max_active_runs=2`.

Required Airflow variables:

| Variable | Meaning |
| --- | --- |
| `BUCKET` | S3 bucket for raw data, jobs, warehouse, logs, and dependencies. |
| `EMR_SERVERLESS_APP_ID` | EMR Serverless application ID. |
| `EMR_JOB_ROLE_ARN` | IAM role used by EMR Serverless jobs. |
| `ATHENA_OUTPUT` | S3 output location for Athena validation query results. |

The DAG has two parallel branches.

Trades branch:

```text
wait_for_trades
  -> ingest_bronze_trades
  -> bronze_to_silver_trades
  -> silver_to_gold_ohlcv
  -> validate_gold

wait_for_events_calendar
  -> bronze_to_silver_trades
```

Quotes branch:

```text
wait_for_quotes
  -> ingest_bronze_quotes
  -> bronze_to_silver_quotes
  -> silver_to_gold_spread
  -> validate_gold_spread
```

Tasks:

| Task | Type | Computes / touches |
| --- | --- | --- |
| `wait_for_trades` | `S3KeySensor` | Waits up to 15 minutes for `raw/trades/dt=<DATE>/hh=<HOUR>/*`. Soft-fails if no data is found. |
| `wait_for_events_calendar` | `ExternalTaskSensor` | Waits for `events_calendar_daily.bronze_to_silver_events` for the same processing day. This guarantees `silver.events` exists before trades are enriched. |
| `ingest_bronze_trades` | EMR Serverless Spark job | Reads raw trade JSONL.GZ files for the current hour and MERGEs into `bronze.trades_raw`. |
| `bronze_to_silver_trades` | EMR Serverless Spark job with PyDeequ | Re-reads the full day from `bronze.trades_raw`, casts, deduplicates, enriches with events, writes through an Iceberg WAP branch, validates, and publishes to `silver.trades`. |
| `silver_to_gold_ohlcv` | EMR Serverless Spark job | Computes 1-minute OHLCV bars from `silver.trades` into `gold.ohlcv_1m`. |
| `validate_gold` | Python/Athena validation | Checks that `gold.ohlcv_1m` has at least one row for the processing date. |
| `wait_for_quotes` | `S3KeySensor` | Waits up to 15 minutes for `raw/quotes/dt=<DATE>/hh=<HOUR>/*`. Soft-fails if no data is found. |
| `ingest_bronze_quotes` | EMR Serverless Spark job | Reads raw quote JSONL.GZ files for the current hour and MERGEs into `bronze.quotes_raw`. |
| `bronze_to_silver_quotes` | EMR Serverless Spark job with PyDeequ | Re-reads the full day from `bronze.quotes_raw`, casts, deduplicates, writes through an Iceberg WAP branch, validates, and publishes to `silver.quotes`. |
| `silver_to_gold_spread` | EMR Serverless Spark job | Computes 1-minute bid/ask spread metrics from `silver.quotes` into `gold.bid_ask_spread_1m`. |
| `validate_gold_spread` | Python/Athena validation | Checks that `gold.bid_ask_spread_1m` has at least one row for the processing date. |

The Bronze-to-Silver trade and quote jobs intentionally read the full processing day every 15 minutes, not just the current hour. That makes late-arriving corrections idempotent: a corrected tick that lands later in the day can update the existing Silver row during the next run.

### `gold_daily`

File: `dags/gold_daily.py`

This DAG computes end-of-day Gold aggregates after the intraday pipeline has completed.

Schedule:

```text
0 23 * * 1-5
```

That means Monday through Friday at 23:00 UTC. `catchup=False`, `max_active_runs=1`.

Required Airflow variables:

| Variable | Meaning |
| --- | --- |
| `BUCKET` | S3 bucket for jobs, warehouse data, and logs. |
| `EMR_SERVERLESS_APP_ID` | EMR Serverless application ID. |
| `EMR_JOB_ROLE_ARN` | IAM role used by EMR Serverless jobs. |
| `ATHENA_OUTPUT` | S3 output location for Athena validation query results. |

Flow:

```text
silver_to_gold_vwap -> validate_vwap
silver_to_gold_event_windowed -> validate_event_windowed
```

Both branches run in parallel.

Tasks:

| Task | Type | Computes / touches |
| --- | --- | --- |
| `silver_to_gold_vwap` | EMR Serverless Spark job | Runs `silver_to_gold.py` with `MODE=vwap`, producing `gold.vwap_daily` from `silver.trades`. |
| `validate_vwap` | Python/Athena validation | Checks that `gold.vwap_daily` has at least one row for the processing date. |
| `silver_to_gold_event_windowed` | EMR Serverless Spark job | Runs `silver_to_gold.py` with `MODE=event_windowed`, producing `gold.event_windowed_volume` from `silver.events` and `silver.trades`. |
| `validate_event_windowed` | Python/Athena validation | Counts rows in `gold.event_windowed_volume` for the processing date. `min_rows=0`, so a day with no events is valid. |

`gold.bid_ask_spread_1m` is no longer computed here. It is computed intraday by `trading_pipeline`.

## Table Catalog

The setup jobs create ten Iceberg tables across three Glue databases: `bronze`, `silver`, and `gold`.

### `bronze.trades_raw`

Created by: `pyspark_jobs/create_bronze_tables.py`

Written by: `ingest_bronze_trades.py`

Purpose: Stores raw trade ticks loaded from hourly JSONL.GZ files. Values are mostly kept in raw form; `exchange_ts` remains a string until Silver. The table is partitioned by `_trade_date`, which comes from the S3 path/date passed by Airflow.

Iceberg properties:

| Property | Value |
| --- | --- |
| Format version | `2` |
| Partitioning | `_trade_date` |
| Write mode | merge-on-read for merge/update/delete |
| Compression | Zstandard Parquet |

Schema:

| Column | Type | Meaning |
| --- | --- | --- |
| `exchange` | `string` | Exchange or venue that produced the trade. |
| `symbol` | `string` | Instrument symbol. |
| `exchange_ts` | `string` | Raw exchange timestamp string from the feed. Cast to timestamp in Silver. |
| `price` | `double` | Raw trade price. |
| `size` | `bigint` | Raw trade size or quantity. |
| `sequence_num` | `bigint` | Exchange sequence number used as part of the natural key. |
| `side` | `string` | Trade side, expected later to be `BUY`, `SELL`, or `MID`. |
| `trade_id` | `string` | Trade identifier from the source feed. Included in the Bronze MERGE key. |
| `_ingest_ts` | `timestamp` | UTC timestamp when the ingest Spark job loaded the row. |
| `_trade_date` | `date` | Operational trade date from the Airflow `DATE` environment variable. Also the Iceberg partition column. |



| `isComplete(symbol)` | Symbol must be present. |
| `isComplete(event_type)` | Event type must be present. |

### `bronze_to_silver_trades`

File: `pyspark_jobs/bronze_to_silver_trades.py`

This task reads all Bronze trades for the processing date:

```sql
SELECT *
FROM glue_catalog.bronze.trades_raw
WHERE _trade_date = DATE '<DATE>'
```

The full-day read is deliberate. Every 15-minute run reprocesses the day so late-arriving corrections can update Silver.

Transform steps:

1. Cast `exchange_ts` from string to timestamp.
2. Derive `trade_date = to_date(exchange_ts)`.
3. Cast `price` to `decimal(18,6)`.
4. Deduplicate on `(exchange, symbol, exchange_ts, sequence_num)` using earliest `_ingest_ts`.
5. Read same-day events:

```sql
SELECT event_id, symbol, event_type, event_date, event_start
FROM glue_catalog.silver.events
WHERE event_date = DATE '<DATE>'
```

6. Deduplicate events to one event per `(symbol, event_date)`, ordered by `event_start`, then `event_id`.
7. Left join trades to events on `symbol` and `trade_date = event_date`.
8. Select columns in Silver DDL order.

Event enrichment logic:

```text
silver_trade.symbol = event.symbol
silver_trade.trade_date = event.event_date
```

WAP write and publish:

```sql
ALTER TABLE glue_catalog.silver.trades CREATE OR REPLACE BRANCH audit_<BRANCH_TS>
```

Then the job sets:

```text
spark.wap.branch = audit_<BRANCH_TS>
```

and MERGEs into the WAP branch:

```sql
MERGE INTO glue_catalog.silver.trades AS t
USING silver_trades_staged AS s
ON  t.exchange    = s.exchange
AND t.symbol      = s.symbol
AND t.exchange_ts = s.exchange_ts
AND t.sequence_num = s.sequence_num
WHEN NOT MATCHED THEN INSERT *
WHEN MATCHED THEN UPDATE SET *
```

It reads back the audit branch for the processing date:

```sql
SELECT * FROM glue_catalog.silver.trades VERSION AS OF 'audit_<BRANCH_TS>'
WHERE trade_date = DATE '<DATE>'
```

It runs PyDeequ checks:

| Check | Purpose |
| --- | --- |
| Required `exchange`, `symbol`, `exchange_ts`, `sequence_num` | Protects the Silver natural key. |
| `side` contained in `BUY`, `SELL`, `MID` | Enforces trade-side domain values. |
| Nonnegative `size` | Prevents invalid quantity. |
| Nonnegative `price` | Prevents invalid price. |
| Uniqueness on `exchange, symbol, exchange_ts, sequence_num` >= 99.99% | Confirms dedup worked and the MERGE key is nearly unique. |

If checks pass, it publishes:

```sql
CALL glue_catalog.system.fast_forward(
  'glue_catalog.silver.trades',
  'main',
  'audit_<BRANCH_TS>'
)
```

Then it drops the audit branch:

```sql
ALTER TABLE glue_catalog.silver.trades DROP BRANCH audit_<BRANCH_TS>
```

If checks fail, it drops the branch and does not publish.

### `silver_to_gold_event_windowed`

File: `pyspark_jobs/silver_to_gold.py`

Mode:

```text

```

This task computes event-window volume and realized volatility.

It first reads events for the processing date:

```sql
SELECT
    event_id,
    symbol,
    event_type,
    event_start AS event_ts
FROM glue_catalog.silver.events
WHERE event_date = DATE '<DATE>'
  AND event_start IS NOT NULL
```

It reads trades from the previous, current, and next date:

```sql
SELECT exchange, symbol, exchange_ts, price, size
FROM glue_catalog.silver.trades
WHERE trade_date BETWEEN DATE '<DATE>' - INTERVAL 1 DAY
                    AND DATE '<DATE>' + INTERVAL 1 DAY
```

The three-day read protects events near midnight UTC. The exact event window still controls inclusion.

It joins trades to events by `symbol`, creates a two-hour window around each event, and filters:

```text
window_start = event_ts - INTERVAL 1 HOUR
window_end   = event_ts + INTERVAL 1 HOUR

exchange_ts >= window_start
exchange_ts <= window_end
```

Volume aggregation:

```text
is_pre = exchange_ts < event_ts
pre_volume = SUM(size where is_pre)
post_volume = SUM(size where not is_pre)
```

Grouped by:

```text
event_ts, event_id, event_type, symbol, window_start, window_end
```

Realized volatility logic:

1. Truncate trades to 1-minute `bar_start`.
2. For each `(symbol, event_ts, bar_start)`, keep the last price by descending `exchange_ts`.
3. For each `(symbol, event_ts)`, compute previous minute price with `lag(price)`.
4. Compute log return:

```text
log_ret = log(price / prev_price)
```

5. Compute:

```text
realized_vol = stddev(log_ret)
```

Finally, it joins volume and realized volatility and selects:

```text
event_ts, event_id, event_type, symbol,
window_start, window_end,
pre_volume, post_volume,
coalesce(realized_vol, 0.0) as realized_vol
```

It MERGEs into Gold:

```sql
MERGE INTO glue_catalog.gold.event_windowed_volume AS t
USING gold_event_windowed_volume AS s
ON  t.event_ts = s.event_ts
AND t.event_id = s.event_id
AND t.symbol = s.symbol
WHEN MATCHED THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *
```

### `validate_event_windowed`

File: `dags/gold_daily.py`

This task runs an Athena row-count query:

```sql
SELECT COUNT(*) AS row_count
FROM gold.event_windowed_volume
WHERE CAST(event_ts AS DATE) = DATE '<DATE>'
```

Unlike other validations, it allows zero rows because a day with no corporate events is valid.

### `run_iceberg_maintenance`

Files:

* `dags/iceberg_maintenance_dag.py`
* `pyspark_jobs/iceberg_maintenance.py`

This task loops over all ten Iceberg tables and runs four maintenance operations per table.

1. Compact small data files:

```sql
CALL glue_catalog.system.rewrite_data_files(
    table => '<TABLE>',
    strategy => 'binpack',
    options => map(
        'target-file-size-bytes', '134217728',
        'min-file-size-bytes',    '33554432',
        'max-concurrent-file-group-rewrites', '5'
    )
)
```

2. Rewrite manifests:

```sql
CALL glue_catalog.system.rewrite_manifests(table => '<TABLE>')
```

3. Expire old snapshots:

```sql
CALL glue_catalog.system.expire_snapshots(
    table => '<TABLE>',
    older_than => TIMESTAMP '<UTC_TIMESTAMP>',
    retain_last => 2
)
```

The DAG passes `SNAPSHOT_MAX_AGE_MS` as seven days.

4. Remove orphan files:

```sql
CALL glue_catalog.system.remove_orphan_files(table => '<TABLE>')
```

The job accumulates errors per table and exits nonzero if any table maintenance fails.

## Operational Patterns

### Idempotent MERGEs

Every ingest and promotion path uses deterministic MERGE keys:

| Table | MERGE key |
| --- | --- |
| `bronze.trades_raw` | `exchange, symbol, exchange_ts, sequence_num, trade_id` |
| `bronze.quotes_raw` | `exchange, symbol, exchange_ts, sequence_num` |
| `bronze.events_raw` | `event_id` |
| `silver.trades` | `exchange, symbol, exchange_ts, sequence_num` |
| `silver.quotes` | `exchange, symbol, exchange_ts, sequence_num` |
| `silver.events` | `event_id, event_date` |
| `gold.ohlcv_1m` | `bar_start, exchange, symbol` |
| `gold.vwap_daily` | `trade_date, exchange, symbol` |
| `gold.bid_ask_spread_1m` | `bar_start, exchange, symbol` |
| `gold.event_windowed_volume` | `event_ts, event_id, symbol` |

This makes reruns safe and lets later corrections update existing rows.

### Iceberg Write Modes

The table write modes match the expected update pattern for each layer:

| Tables | Write mode | Reason |
| --- | --- | --- |
| `bronze.trades_raw`, `bronze.quotes_raw`, `bronze.events_raw` | Merge-on-read | Bronze ingest is correction-friendly: reruns and re-deliveries can update existing natural-key rows without rewriting whole data files immediately. |
| `silver.trades`, `silver.quotes` | Merge-on-read with WAP | These are high-volume tick tables refreshed repeatedly during the day. Merge-on-read keeps frequent MERGEs practical, while WAP prevents invalid data from being published to `main`. |
| `silver.events` | Copy-on-write | Events are low-volume reference data. Rewriting affected data files is acceptable, and the table does not need the extra WAP branch workflow used for high-volume tick data. |
| Gold tables | Copy-on-write | Gold tables are smaller because they aggregate many Silver rows into fewer analytical rows, and they are the primary consumption layer for Athena and analytics queries. Copy-on-write keeps that layer clean while keeping rewrite cost reasonable. |

In short, merge-on-read is used where mutable, high-frequency ingest performance matters most. Copy-on-write is used for Gold because the tables are smaller aggregate outputs and because Gold is the user-facing analytical layer, where clean reads are more important than optimizing frequent row-level mutations.

### Write-Audit-Publish for High-Volume Silver Tables

`silver.trades` and `silver.quotes` use Iceberg WAP branches:

1. Create an `audit_<BRANCH_TS>` branch.
2. Write the MERGE into that branch.
3. Read the branch back.
4. Run PyDeequ checks.
5. Fast-forward `main` only if checks pass.
6. Drop the audit branch.

This prevents invalid high-volume tick data from being published to the main Silver branch.

### Data Quality Gates

PyDeequ validates Silver outputs for trades, quotes, and events. Athena validates Gold row counts after aggregation tasks. Together they catch both row-level quality issues and missing aggregate partitions.

### UTC Processing Dates

The DAGs use `data_interval_end` for templated dates and hours. Spark sessions set:

```text
spark.sql.session.timeZone = UTC
```

This prevents timestamp casts from depending on container-local JVM timezones.

### Maintenance

The weekly maintenance DAG keeps Iceberg tables healthy by:

* Compacting small data files.
* Rewriting manifests.
* Expiring snapshots older than seven days while retaining the last two.
* Removing orphan files.
